# Wazuh Home Lab — Agent Setup & Troubleshooting (Part 2)
**May 28, 2026**

---

## Overview

This document covers the second phase of the lab build — connecting the Windows 10 VM to the Wazuh server as a monitored endpoint. This phase involved enabling SSH, troubleshooting agent installation, and resolving agent registration conflicts.

---

## Step 1 — Accessing the Wazuh Dashboard

With the Wazuh server running on Ubuntu, the dashboard was accessed from the Windows 10 VM browser at:

```
https://192.168.56.103
```

A browser security warning appeared due to a self-signed certificate — this is expected. Clicked **Advanced → Proceed** to access the dashboard. Logged in with username `admin` and the generated password from installation.

Dashboard loaded successfully showing 121 medium and 157 low severity alerts already generated by the server itself. Agents Summary showed 'No results' — expected since no endpoints were connected yet.

---

## Step 2 — Installing the Wazuh Agent on Windows 10

Attempted to install the Wazuh agent on the Windows 10 VM via PowerShell:

```powershell
Invoke-WebRequest -Uri https://packages.wazuh.com/4.x/windows/wazuh-agent-4.9.2-1.msi -OutFile wazuh-agent.msi
.\wazuh-agent.msi /q WAZUH_MANAGER="192.168.56.103"
```

First attempt failed silently — the MSI downloaded but the agent service (WazuhSvc) did not appear. Reinstalled using msiexec directly which worked:

```powershell
msiexec /i wazuh-agent.msi /q WAZUH_MANAGER="192.168.56.103"
```

## Problems Encountered & Fixes

### 1. Service Failing to Start — Invalid Server Address

The agent service kept stopping immediately after starting. Checked the agent log:

```powershell
Get-Content "C:\Program Files (x86)\ossec-agent\ossec.log" -Tail 30
```

Error showed: `Invalid server address found: 0.0.0.0` — the manager IP was not written to the config during installation. Fixed by manually editing the config:

```powershell
notepad "C:\Program Files (x86)\ossec-agent\ossec.conf"
```

Changed `<address>0.0.0.0</address>` to `<address>192.168.56.103</address>` and saved.

Verified the change saved correctly:

```powershell
Get-Content "C:\Program Files (x86)\ossec-agent\ossec.conf" | Select-String "address"
```

---

### 2. Enabling SSH for Easy Copy/Paste

Clipboard sharing between VMs was unreliable. Solved by enabling SSH on Ubuntu so Windows PowerShell could connect directly:

```bash
sudo apt install openssh-server -y
sudo systemctl enable ssh
sudo systemctl start ssh
```

Connected from Windows PowerShell:

```powershell
ssh vboxuser@192.168.56.103
```

---

### 3. Agent Registering But Never Connecting

The agent showed as 'Never connected' on the server even though the service appeared to start. Tested network connectivity between VMs:

```powershell
Test-NetConnection -ComputerName 192.168.56.103 -Port 1514
Test-NetConnection -ComputerName 192.168.56.103 -Port 1515
```

Both ports returned `TcpTestSucceeded: True` — network was fine. Checked Ubuntu firewall:

```bash
sudo ufw status
```

UFW was inactive — firewall was not the issue. Opened ports anyway as precaution:

```bash
sudo ufw allow 1514/tcp
sudo ufw allow 1515/tcp
sudo ufw allow 1514/udp
```

Used tcpdump to monitor traffic — confirmed no Wazuh traffic was reaching the server, only SSH traffic:

```bash
sudo tcpdump -i enp0s8
```

---

### 4. Manual Agent Registration

Used the Wazuh agent manager tool on Ubuntu to manually register the Windows endpoint:

```bash
sudo /var/ossec/bin/manage_agents
```

Added agent with name `windows10` and IP `192.168.56.102`, assigned ID 001. Extracted the registration key and imported it on Windows:

```powershell
& "C:\Program Files (x86)\ossec-agent\manage_agents.exe" -i <key>
```

Checked agent status on Ubuntu:

```bash
sudo /var/ossec/bin/agent_control -l
```

---

### 5. Duplicate Agent Name Conflict

After a clean reinstall of the Windows agent, logs revealed the root cause:

The manual registration from the previous step conflicted with the auto-registration from the fresh install. Fixed by removing the old registration on Ubuntu:

```bash
sudo /var/ossec/bin/manage_agents
```

Selected `R` to remove, entered ID `001`, confirmed with `y`, then quit. Restarted the manager:

```bash
sudo systemctl restart wazuh-manager
```

Restarted the Windows agent service — it auto-registered cleanly this time.
